# HK Pre-owned Rolex Monitoring 项目目标与实施顺序

版本：v3
日期：2026-07-14
状态：当前执行基线

## 一、总目标

先完成一个真正平台中立、可独立运行的 HK Pre-owned Rolex Monitoring Skill；再将它作为公共项目发布到 GitHub；最后为云端 QClaw 提供一份可以直接复制使用的调用 PE。

新的实施顺序是：

```text
完整 Skill
→ 多种 IDE / Agent 可调用
→ GitHub 公共发布与项目展示
→ 云端 QClaw 复制 PE 调用
```

QClaw 不再是 Skill 的首要设计前提，只是完成 Skill 和 GitHub 发布后的一个调用宿主。

## 二、第一阶段：完成平台中立的 Skill

### 阶段目标

完成 `hk-pre-owned-rolex-monitoring` Skill 本体，使它专门监控香港东方表行 Rolex CPO，同时不依赖 QClaw、特定 IDE、聊天记录或私有运行环境。

### 必须完成

1. 创建规范的 Skill 目录，至少包含：
   - `SKILL.md`；
   - `agents/openai.yaml`；
   - 确定性执行脚本；
   - 必要的 `references/`；
   - 必要的 `assets/`；
   - 自动化测试和离线 Fixture。
2. 提供稳定的命令行入口，让支持执行命令的 IDE 或 Agent 都能调用。
3. 所有关键操作提供结构化 JSON 输出，不要求宿主解析自然语言日志。
4. 核心逻辑保持平台中立，不导入 QClaw、Codex、Cursor、Claude Code 或其他宿主的私有模块。
5. 完成库存监控核心能力：
   - 创建与校验 Monitor Manifest；
   - 建立可信基线；
   - 生成稳定商品 ID；
   - 检测上新、下架、价格变化和资料变化；
   - 每条变化提供商品名称、Rolex 型号编号和东方表行货号，不把内部 stable ID 单独展示给用户；
   - 成功基线和成功运行期间把商品图片下载到独立状态目录；下架后保留历史图片，并在变化与 Outbox 中返回可用缓存路径；
   - 图片缓存失败只产生警告，不污染库存判断；云端宿主必须验证状态目录跨重启持久；
   - 识别不可信抓取并返回 `INVALID`；
   - 保证异常结果不覆盖成功基线；
   - 运行锁、幂等、通知事件队列、备份、恢复和状态迁移。
6. 完成东方表行 Rolex CPO 首个 Site Adapter，并使用脱敏 Fixture 测试。
7. 为不同宿主提供统一调用说明：
   - 原生支持 Agent Skill 的宿主读取 `SKILL.md`；
   - IDE Agent 通过终端调用 CLI；
   - 普通开发环境直接运行 CLI。
8. 提供独立的二手 Rolex 市场情报层：
   - 提供机器可读的来源政策注册表，记录自动化状态、凭证、license、条款核验日期、速率和存储边界；
   - 提供 `market source doctor`，在请求前明确返回凭证缺失、license 不足、自动化禁止或条款待审查；
   - 首个自动行情来源只使用 WatchCharts 官方 API；API key 和 license 声明由用户提供，不写入仓库或 CLI 输出；
   - Wristcheck、Chrono24 及尚未确认自动化权限的来源不得默认网页抓取，只接受人工证据、授权导出或取得书面许可后的未来 Adapter；
   - 支持带来源证据的香港、亚太、大陆和全球 Market Packet；
   - 区分 `fixture`、`unverified` 和 `verified` 证据，示例或未核验数据不得形成已验证参考价；
   - 行情比较以 Rolex 型号和目标年份窗口为核心，默认比较前后 2 年，并优先采用好成色样本；
   - 每个独立来源先按年份计算均价，再跨年份、跨独立来源求均值，避免单一平台或单一年份样本量支配结果；
   - 模型估值、交易指数、挂牌价、拍卖结果和商户报价必须按地区与价格口径分组，不得混合求一个参考价；
   - 同一地区与价格口径组内至少两个独立可信来源才形成已验证参考价；
   - 输出面向用户的中文行情摘要，并显式说明型号、年份窗口、主参考 cohort、证据数和参考价差异；
   - 市场数据无效或不足不得修改库存基线，也不得被描述成确定性买卖建议。
9. 提供平台中立的运行环境 Bootstrap：
   - 在第三方依赖安装前使用纯标准库检查 Python、pip、真实依赖下载、安装权限、PATH 和状态目录；
   - 包索引首页可访问不得等同于依赖可下载；默认源、配置镜像、DNS、TLS 和权限错误分别报告；
   - 镜像只能由用户或宿主通过环境配置，不在 Skill 中硬编码或静默切换；所有镜像凭证必须脱敏；
   - 生产环境使用固定 Release wheel 和 SHA-256，优先独立虚拟环境，必要时回退可写的用户目录；
   - 安装后必须实际运行 `skill info` 和可选的 `runtime probe`，只有结构化回执为 `INSTALL_VERIFIED` 才算完成。

### 多 IDE / Agent 兼容性的定义

“各种 IDE 可以调用”不表示所有 IDE 使用完全相同的安装按钮，而是指：

- Skill 的业务核心与宿主无关；
- 仓库提供标准 `SKILL.md` 和稳定 CLI；
- 任何能够读取文件、执行命令、访问网络并保存状态的 IDE Agent，都可以接入；
- 宿主必须把 `state-dir` 放在持久存储中；一次目录写入成功不能等同于已经验证跨重启持久；
- 对不具备定时或通知能力的宿主，Skill 返回声明式 Runtime Plan 或明确的能力缺失结果；
- 不允许为适配某个 IDE 而复制或改写核心库存算法。

### 第一阶段完成标准

- Skill 结构校验通过；
- 核心测试、Adapter Fixture 测试和 CLI 测试通过；
- 全新目录中可以安装并运行；
- 至少完成通用终端调用验证和可用宿主的调用验证；
- 未验证的 IDE 必须标记为“兼容设计已完成、尚未实测”，不得声称全部验证完成；
- 未完成第一阶段前，不开始把 QClaw 逻辑写进 Skill。

## 三、第二阶段：GitHub 公共发布与项目封面

### 阶段目标

将第一阶段完成的 Skill 发布为任何人都能访问、理解、安装和复用的 GitHub 公共项目。

### 必须完成

1. 创建 GitHub Public repository。
2. 确定 License、仓库名称、版本号和最低运行环境。
3. 提供清楚的公共 README，包括：
   - 项目用途；
   - 主要能力；
   - 安装方法；
   - 快速开始；
   - IDE / Agent 调用方式；
   - 更新、备份、恢复和卸载；
   - 安全边界和已知限制。
4. 完成项目视觉包装：
   - GitHub README 顶部封面图；
   - Skill 图标；
   - 统一名称、颜色和一句话介绍；
   - 必要的运行示例或效果图。
5. 在设计封面和 README 前，参考优秀公开 Skill 的信息结构、视觉层次和安装说明，但不复制其品牌或受保护素材。
6. 配置 CI、Issue 模板、贡献说明、安全说明和 Release 流程。
7. 发布首个正式版本，并提供固定 tag、Release notes、校验信息和回滚说明。
8. 使用 GitHub Release 在全新环境执行 clean-install 验证，不能依赖开发者本机路径。

### 第二阶段完成标准

- 仓库可以匿名访问和克隆；
- README、封面、图标和示例完整；
- CI 全部通过；
- 存在正式 Release，而不是只提供 `main`；
- Release 中不包含 Token、Cookie、私人库存、通知标识或用户状态；
- 陌生用户只看 README 就能完成安装和首次运行；
- 能提供真实 repository URL、Release URL、tag 和 commit SHA。

## 四、第三阶段：交付云端 QClaw 可复制 PE

### 阶段目标

为云端 QClaw 提供一套可以直接复制粘贴的 PE，使 QClaw 从 GitHub 正式 Release 安装、读取并调用 HK Pre-owned Rolex Monitoring Skill。

### 交付物

至少提供：

1. 一份整合版 PE：适合用户第一次直接复制到 QClaw；
2. Bootstrap PE：负责安装、版本锁定、能力检查和初始化；
3. Invoke PE：负责日常运行、状态检查和通知；
4. Repair PE：负责诊断、重建任务、升级和回滚；
5. 精简定时任务正文模板。

整合版 PE 必须已经写入：

- 真实 GitHub 公共仓库地址；
- 正式 Release 版本；
- Skill 名称；
- 稳定调用入口；
- 需要 QClaw 验证的能力；
- 明确的成功、失败和未支持报告格式。

### 云端 QClaw 的特殊要求

- PE 不能引用本地电脑路径；
- QClaw 必须从公共 GitHub Release 获得 Skill；
- 默认锁定正式 Release，不能每次运行都拉取 `main`；
- 用户状态必须保存在 QClaw 可持久化的位置，并与 Skill 安装目录分离；
- PE 只负责安装、调用、定时、通知和验证，不能复制库存核心算法；
- 不猜测 QClaw 的命令、API、任务 ID 或持久化能力；
- 如果 QClaw 不支持某项能力，PE 必须明确报告，不得虚报已经部署。

### 第三阶段完成标准

用户把整合版 PE 复制到云端 QClaw 后，QClaw应当能够：

```text
获取指定 GitHub Release
→ 安装并读取 SKILL.md
→ 检查运行能力
→ 创建 Monitor Manifest
→ 建立可信基线
→ 执行 Skill 返回的 Runtime Plan
→ 创建并验证定时任务
→ 保存真实任务 ID
→ 测试通知
→ 输出验收报告
```

如果由于 QClaw 实际能力或权限无法完成其中某一步，PE 必须停在安全状态，并准确告诉用户缺少什么。

## 五、阶段门禁

- 第一阶段没有通过测试和 clean-install，不进入 GitHub 正式发布。
- 第二阶段没有公共仓库和正式 Release，不交付声称可运行的云端 QClaw PE。
- 第三阶段 PE 没有经过真实 QClaw 演练，只能标记为“候选版、待实测”。
- 任何阶段都不能把设计目标写成已经完成的执行结果。

## 六、当前执行起点

当前从第一阶段开始，优先完成以下最小闭环：

1. 初始化标准 Skill；
2. 定义平台中立契约和稳定 CLI；
3. 使用离线 Fixture 建立两次一致的测试基线；
4. 生成一次上新、下架、改价和 `INVALID` 测试结果；
5. 验证异常运行不会污染成功基线；
6. 完成 Skill 结构校验、自动化测试和全新环境运行验证。

完成这个闭环后，再进入 GitHub 公共发布和封面设计。
